Этот репозиторий содержит мое решение для ML-соревнования Data Fusion Contest 2026 - задача 1 «Страж».

Описание задачи организатором сохранено в `docs/`.

Задача соревнования - детекция мошеннических транзакций. Главная метрика - PR-AUC (average precision).

Датасет - набор пользовательских транзакций с признаками. Хронологически первая половина данных не размечена по таргету и используется для агрегатов и контекста.

Стек: сборка датасета на **C++ + Apache Arrow/Parquet** (`build_dataset`, `build_global_aggregates`), потоковое чтение больших parquet без полной загрузки в память. Обучение и инференс - **Python** с **XGBoost**. Conda-окружения лежат в `requirements/`.

Сборка датасета была перенесена на C++, потому что с ростом числа фич Python-код перестал выполняться в разумное время.

Признаки в модели двух типов: (1) локальные - скользящее окно по пользователю (до 512 транзакций плюс календарные куски 1h / 6h / 24h / 5d / 7d и др.); (2) глобальные - агрегаты по осям MCC, канал, timezone × currency, event_type_nm × currency, event_descr, и др, список имён - в `docs/cpp_dataset_features.md`.

В своём решении я обучал модель XGBoost на большом наборе признаков (244), составленном из пользовательских агрегатов и соответствующих текущей транзакции глобальных агрегатов.

Я обнаружил, что распределение числа транзакций на одного пользователя в данных для посылки сильно отличается от того же в тренировочных данных, из-за чего модель обучалась на более стабильных агрегатах, и метрика на валидации не совпадала с метрикой посылки. Принял решение собрать распределение длины контекста с инференса и семплить на его основе размер окна при обучении. Так была решена проблема сдвига контекста train и inference.

В процессе работы для улучшения качества и отбора признаков я:

1) запускал grid search по гиперпараметрам XGBoost;
2) сверял важность фич по permutation на той же метрике;
3) сравнивал разные бустинг-модели;
4) пробовал реализовать RNN/трансформер по части контекстных фич, однако результаты показали сильное превосходство бустинга на полных агрегатах в этой задаче.

Итоговое значение метрики area under precision-recall curve во время соревнования составило **0.1**. После окончания соревнования, добавив глобальные агрегаты, я поднял офлайн-качество примерно до **0.14**. Целевой класс очень редкий (порядка **0.1%** положительных в train), и сравнивая со случайным предсказанием по доле позитивов, прирост метрики при использовании моей модели составляет порядка сотни раз.

Структура: `dataset_cpp/` - сборка parquet-датасета и глобальных агрегатов, `training/` - обучение бустинга и grid search, `submission/` - инференс (посылка). Окружения: `requirements/environment_cpp.yml` (CMake + Arrow для сборки бинарников) и `requirements/environment_python.yml` (обучение и генерация сабмита).

## Запуск

`conda env create -f requirements/environment_cpp.yml` и `conda env create -f requirements/environment_python.yml`, скачайте данные соревнования и положите их в `data/` так, как подписаны .md заглушки в `data/`

```bash
conda activate guard-cpp
cd dataset_cpp && cmake -B build -S . -DCMAKE_BUILD_TYPE=Release && cmake --build build -j
./build/build_dataset .. && ./build/build_global_aggregates .. && cd ..
conda activate guard-contest
python -m training.main
python -m submission.main
```
