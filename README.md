Этот репозиторий содержит мое решение для ml соревнования Data Fusion Contest 2026 — задача 1 «Страж».

Описание задачи организатором сохранено в `docs/`.

Задача соревнования - детекция мошеннических транзакций. Главная метрика - PR-AUC (average precision).

Датасет - набор пользовательских транзакций с признаками. Хронологически первая половина данных не размечена по таргету и используется для агрегатов и контекста.

Стек: сборка датасета на **C++ + Apache Arrow/Parquet** (`build_dataset`, `build_global_aggregates`), потоковое чтение больших parquet без полной загрузки в память. Обучение и инференс - **Python** с **XGBoost**. Два conda-окружения (C++ и Python) лежат в `requirements/`.

Сборка датасета была перенесена на си++, потому что с ростом числа фич питон код перестал выпоплняться в разумное время.

Признаки в модели двух типов: (1) локальные - скользящее окно по пользователю (до 512 транзакций плюс календарные куски 1h / 6h / 24h / 5d / 7d и др); (2) глобальные - агрегаты по осям MCC, канал, timezone×валюта, event_type_nm×валюта, event_descr, pos_cd и joint-счётчики. Список имён - в `docs/cpp_dataset_features.md`.

В своем решении я обучал модель XGBoost на большом наборе признаков (244), составленным из пользовательских аггрегатов и соответствующих текущей транзакции глобальных аггрегатов. 

Я обнаружил, что распределение числа транзакций на одного пользователя в данных для посылки сильно отличается от того же в тренировочных данных, из-за чего модель обучалась на более стабильных аггрегатах, и метрика на валидации не соответствовала с метрикой посылки. Принял решение собрать распределение длины контекста с инференса и семплить на его основе размер окна при обучении. Так была решена проблема сдвига контекста train и inference.

В процессе работы для улучшения качества и отбора признаков я:
1) запускал grid search по гиперпараметрам XGBoost
2) сверял важность фич по permutation на той же метрике;
3) сравнивал разные бустинг-модели;
4) пробовал реализовать RNN/трансформер по части контекстных фич, однако результаты показали сиьное превосходство бустинга на полных аггрегатах в этой задаче.

Итоговое значение метрики area under precision-recall curve во время соревнования составило **0.1**. После окончания соревнования, добавив глобальные агрегаты, я поднял офлайн-качество примерно до **0.14**. Целевой класс очень редкий (порядка **0.1%** положительных в train), и сравнивая со случайным предсказанием по доле позитивов, прирост метрики при использвоании моей модели составляет порядка сотни раз.

Структура: `dataset_cpp/` - сборка parquet-датасета и глобальных агрегатов, `training/` - обучение бустинга и грид-серч, `submission/` - инференс (посылка).

В `requirements/` лежат окружения для сборки C++ и для питон-обучения модели и посылки в контест.
