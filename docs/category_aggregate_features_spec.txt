Спецификация: агрегаты и признаки по категориальным осям (офлайн-профили, соревнование)
================================================================================
Версия: черновик для модернизации пайплайна (объединена с практическим чеклистом).
         Фокус документа — только глобальные (популяционные) таблицы по категориям;
         pretrain + train по правилам п.6.
Оси: mcc_code; channel_indicator_type + channel_indicator_subtype; timezone;
     event_type_nm


--------------------------------------------------------------------------------

Глобальных (популяционных) агрегатов по mcc_code / каналу / всем клиентам в этом C++-коде
нет — только статистики внутри истории одного пользователя (и локальные срезы по времени).
Вывод для модернизации: приоритет новой работы — разделы B, D.0 и популяционные C–F (join
глобальных таблиц к строкам train/test).

Общие правила (обязательно учитывать при реализации)
--------------------------------------------------------------------------------
0) Глобальная статистика по категориям (разделы B, D.0 и популяционные блоки C–F):
   Таблицы по mcc_code, channel+subtype, timezone, event_type_nm и т.п. строятся офлайн
   по схеме из п.6: сначала накопление на pretrain, затем уточнение на train с правилами
   по мошенническим операциям. Без привязки «только прошлое относительно строки» внутри
   корпуса — сознательное упрощение под соревнование.
   Для инференса на test / pretest те же таблицы замораживают по итогу pretrain+train
   (без test), без подмешивания тестовых операций в агрегаторы.


1) Разреженность: для редких ключей (редкий MCC, редкая пара канал+подтип) —
   сглаживание к «глобальной» базе (backoff), минимальный порог n_obs, или иерархия
   (subtype → type → ALL).

2) Стабильность ключа: нормализовать строки ключа (trim, единый регистр где уместно);
   пустой ключ — отдельный бакет MISSING или исключение из популяционного сравнения.

3) Валюта: агрегаты по сумме — в первую очередь внутри одной currency_iso_cd; при
   смешении валют в одном ключе — либо только счётчики/доли без суммы, либо конвертация
   по курсу (если появится в данных).

4) Меток для претреин файлов нет. Частоты мошенничества по категориям
   и прочие target-зависимые агрегаты накапливаются только на train по правилам п.6
   (pretest/pretrain без разметки в них не участвует). На test подставляются замороженные
   значения из итоговых таблиц. Если нужна строгая каузальная валидация по времени —
   отдельная ветка кода (префикс train до cut-off).

5) Уровень пользователя (окно deque, MCC-фичи внутри клиента и т.д.) — в dataset_cpp.
   Этот документ описывает только популяционный слой: насколько операция типична для
   категории (mcc_code, канал+подтип, …); таблицы по п.6 (pretrain + train).

6) Два прохода для глобальных таблиц (pretrain → train) и определение «мошенничества»
--------------------------------------------------------------------------------
   Операция считается мошеннической (для правил ниже) по наличию её event_id в файле
   train_labels.parquet: если event_id есть в train_labels — мошенничество; если нет —
   не мошенничество.

   Фаза 1 — проход по претренировочным файлам (pretrain_part_*, без разметки фрод/не фрод):
   - Накапливается вся «обычная» глобальная статистика по категориям: счётчики операций,
     суммы, средние, квантили, доли каналов/TZ/типов и т.д. — всё, что не требует метки
     мошенничества.
   - Агрегаты, завязанные на частоту фрода (fraud_rate_mcc, fraud_count, WOE и аналоги),
     на pretrain НЕ обновляются: в этих данных нет разделения фрод/не фрод по train_labels.

   Фаза 2 — проход по тренировочным файлам (train_part_*), с доступом к множеству event_id
   из train_labels:
   - «Чистый» профиль категории (типичные суммы, распределения, счётчики без фрода):
     обновлять только по транзакциям, которые НЕ мошеннические — т.е. у которых event_id
     отсутствует в train_labels. Так типичный чек и прочие marginals не размываются
     аномальными операциями из размеченного подмножества.
   - Частота мошенничества по категории: на этом же проходе по train отдельно накапливать
     статистику для оценки доли/числа мошеннических операций в разрезе каждого ключа
     категории (например число train-операций с данным mcc_code, из них сколько с id в
     train_labels — для fraud_rate и производных). Здесь используются все релевантные
     train-строки (и с меткой в train_labels, и без), в зависимости от определения
     числителя и знаменателя (стандартно: fraud_count_key / total_train_count_key или
     сглаженный вариант).

   Итоговые таблицы для join к строкам датасета = состояние после фазы 1 + фазы 2.
   На test/pretest для инференса подмешивать только эти замороженные таблицы, без второго
   прохода по тесту в агрегаторы.

Нотация ниже:
  Префикс global_* / «глобальные таблицы» = статистика по ключу категории после фаз
  pretrain + train по правилам п.6; одна и та же строка таблицы для всех операций с этим
  ключом при обучении; на test — join к замороженной таблице.
  «Производные фичи» — что считать для строки, сравнивая текущие поля с глобальным профилем.


================================================================================
B. MUST HAVE: глобально по mcc_code (популяция, без customer_id в ключе)
================================================================================
Режим соревнования (п.0, п.6): сначала проход pretrain — базовые marginals по категориям;
затем проход train — донастройка «чистых» статистик только по не-мошенническим строкам
(event_id ∉ train_labels) и отдельное накопление fraud-частот по категориям. Итог —
словарь/таблица, join по ключу для train и test. Внутри train допускается lookahead
относительно порядка строк; на test таблица заморожена.

B.1. Базовые распределения суммы mcc_code
--------------------------------------------------------------------------------
  «Типичный чек» в категории:
  - global_mean_amount_mcc, global_std_amount_mcc, global_median_amount_mcc
  - global_q25_mcc, global_q75_mcc, global_q95_mcc
  (q50 ≈ median; при необходимости p90/p99 — уже частично в разделе C)

B.2. Частотные / охват
--------------------------------------------------------------------------------
  - global_cnt_mcc — число операций с этим MCC: обычно pretrain (все) + train (все), для
    охвата и редкости; при необходимости дублируют отдельным счётчиком только те строки,
    что вошли в расчёт mean/std (pretrain все + train без id в train_labels).


B.3. Отклонение текущей операции от глобальной нормы MCC
--------------------------------------------------------------------------------
  Сильные фичи:
  - amount / global_mean_amount_mcc
  - global_zscore_mcc = (amount - global_mean_amount_mcc) / (global_std_amount_mcc + eps)
  (аналоги относительно median / mad — см. раздел C, не дублировать в коде под разными
  именами без нужды)

B.4. Редкость MCC
--------------------------------------------------------------------------------
  - inv_global_cnt_mcc или 1 / (global_cnt_mcc + c) — сглаживание обязательно


B.5. «Рисковость» MCC по таргету
--------------------------------------------------------------------------------
  Считается только на train-фазе (п.6): fraud_rate_mcc, fraud_count_mcc, woe_mcc и т.д.
  Определение мошеннической операции — как в п.6 (event_id ∈ train_labels).
  «Типичные» суммы/квантили по MCC при п.6 обновляются на train без учёта таких строк;
  fraud-частоты — отдельные счётчики по всем train-операциям в ключе (см. п.6).
  На test меток нет — подставляются замороженные значения таблиц.
  Для строгой временной валидации — отдельные таблицы только на префиксе train.

B.6. Стабильность сумм по MCC
--------------------------------------------------------------------------------
  - global_cv_mcc = global_std_amount_mcc / (global_mean_amount_mcc + eps)
  Интерпретация: «стабильные» категории vs с большим разбросом (travel, luxury и т.д.)




C.1. Канал и способ проведения (условное распределение внутри MCC)
--------------------------------------------------------------------------------
  Базовые агрегаты:
  -доли channel_indicator_type и channel_indicator_subtype 

  Производные фичи:
  - редкость канала для mcc_code
 


C.2. Валюта и география времени
--------------------------------------------------------------------------------
  Базовые агрегаты по MCC:
  - доли currency_iso_cd -> фича частоты
  - доли timezone  -> фича частоты




================================================================================
D. channel_indicator_type + channel_indicator_subtype (составной ключ)
================================================================================


D.0. MUST HAVE: глобальные агрегаты по каналу (популяция)
--------------------------------------------------------------------------------
  Как раздел B: таблица строится офлайн по pretrain+train (п.6), один join по ключу
  channel_type + channel_indicator_subtype.
  Семантика: «как обычно совершается операция в этом канале / подтипе» (web, mobile,
  atm, pos, browser vs app и т.д.).
  Базовые:
  - global_mean_amount_channel, global_std_amount_channel, global_median_amount_channel
  - квантили суммы: q50, q75, q95 (или согласовать с global_q* в разделе B)
  - global_cv_channel = std / (mean + eps)
  Отклонение для текущей транзакции:
  - amount / global_mean_amount_channel
  - zscore_channel = (amount - global_mean_amount_channel) / (global_std_amount_channel + eps)
  Если доступен таргет:
  - fraud_rate_channel, fraud_rate_channel_subchannel (или единый ключ type+subtype)



D.1. Суммы (внутри channel_key × currency)
--------------------------------------------------------------------------------
  Базовые: median, p90, p99, std/mad, trimmed mean.

  Производные: по аналогии с B.1–B.3, но относительно профиля канала: amount_z_vs_channel_median,
  amount_percentile_in_channel.

D.2. MCC и тип операции внутри канала
--------------------------------------------------------------------------------
  Базовые: распределение mcc_code  event_type_nm;

  Производные: surprise_mcc_given_channel, surprise_event_type_given_channel,
  mcc_not_in_channel_top3_flag.



================================================================================
E. timezone
================================================================================
Ключ: timezone (MISSING отдельно). Осторожно: редкие TZ → backoff к REGION или ALL.


E.1. Суммы (timezone)
--------------------------------------------------------------------------------
  Базовые: median_amount, p25, p75 по TZ×валюта.

  Производные: amount_z_vs_tz_median; amount_percentile_in_tz_currency.

E.2. Время суток (локально к TZ)
--------------------------------------------------------------------------------
  Базовые: гистограмма часа операции (0–23) в TZ; доля ночи/утра/вечера; пиковый час.

  Производные: surprise_hour_given_tz (текущий локальный час vs распределение);


E.3. Канал и MCC в TZ
--------------------------------------------------------------------------------
  Базовые: топ MCC, топ channel_subtype, энтропии.

  Производные: surprise_mcc_given_tz, surprise_channel_given_tz.

E.4. Устройство / язык
--------------------------------------------------------------------------------
  Базовые: доли OS, accept_language, browser_language.

  Производные: surprise_os_given_tz; accept_lang_vs_tz_majority_mismatch.


================================================================================
F. event_type_nm
================================================================================
Ключ: event_type_nm (MISSING отдельно). Часто меньше уникальных значений, чем MCC —
хорошая ось для устойчивых профилей.

F.1. Объём
--------------------------------------------------------------------------------
  Производные: global_type_frequency_log; редкость типа в популяции (например -log доли).

F.2. Суммы (event_type_nm × currency)
--------------------------------------------------------------------------------
  Базовые: median, p90, p99, std/mad.

  Производные: amount_z_vs_event_type_median; amount_percentile_in_event_type.

F.3. MCC и канал при фиксированном типе операции
--------------------------------------------------------------------------------
  Базовые: распределение mcc_code; распределение channel_type/subtype; энтропии.

  Производные: surprise_mcc_given_event_type; surprise_channel_given_event_type;
  mcc_channel_joint_surprise (опционально).

F.4. Время и сессии
--------------------------------------------------------------------------------
  Базовые: медиана интервала между операциями этого типа; доля ночных; среднее число
  операций этого типа на сессию.

  Производные: gap_vs_event_type_typical; night_flag vs type_night_baseline.
