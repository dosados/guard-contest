================================================================================
Пайплайн: популяционные агрегаты по категориям (C++ → файл → train / инференс)
================================================================================
Назначение: зафиксировать целевую логику реализации согласованно со
`docs/category_aggregate_features_spec.txt` и с требованием быстрого обучения
(агрегаты в train-parquet уже материализованы; на инференсе — подгрузка того
же артефакта в Python).


--------------------------------------------------------------------------------
1. Цели и разделение train vs inference
--------------------------------------------------------------------------------

Обучение (XGBoost):
  - Чтение `output/full_dataset.parquet` должно оставаться последовательным и
    без тяжёлого join в Python: все производные фичи от популяционных таблиц
    уже лежат в колонках parquet на этапе сборки датасета в `dataset_cpp`.

Инференс (submission, Python):
  - В момент прохода по pretest/test глобальные таблицы не пересчитываются.
  - Один раз загружаются с диска те же агрегаты, что использовались при сборке
    train-датасета (заморозка после pretrain+train по спеке, без test).
  - Для каждой строки после формирования «локальных» фич из `compute_features`
    (или внутри того же слоя) добавляются значения из таблиц по ключам строки
    (mcc_code, канал+подтип, timezone, event_type_nm, …).


--------------------------------------------------------------------------------
2. Формат файла с агрегатами (рекомендация)
--------------------------------------------------------------------------------

Формат: Apache Parquet (.parquet).

Почему:
  - Уже используется в проекте (train-датасет, pyarrow в training/submission).
  - Колоночное хранение, сжатие, быстрое чтение только нужных колонок.
  - Один бинарник C++ (Arrow C++) может писать те же типы, что читает Python.


    output/global_aggregates/mcc.parquet
    output/global_aggregates/channel_subtype.parquet
    output/global_aggregates/timezone.parquet
    output/global_aggregates/event_type_nm.parquet
  Каждый файл: одна строка на значение ключа (+ опционально составные ключи
  как строка или несколько колонок), далее столбцы с числовыми агрегатами.


Имена колонок в таблицах агрегатов должны совпасть с теми именами, которые
попадут в `FEATURE_NAMES` / `MODEL_INPUT_FEATURES` для глобального блока фич
(единый источник правды — список в коде, синхронно с C++ при записи в parquet).


--------------------------------------------------------------------------------
3. Этап 1: C++ — построение агрегатов и запись в файл(ы)
--------------------------------------------------------------------------------

Отдельная логика (отдельный исполняемый модуль или явно выделенная фаза в том же
проекте, что и `dataset_cpp`):

Входы:
  - `data/train/pretrain_part_*.parquet`
  - `data/train/train_part_*.parquet`
  - `data/train_labels.parquet` (множество event_id для правил «мошенничество»)

Алгоритм (соответствие спецификации, п.6):
  - Фаза 1: проход только pretrain — накопление «обычной» популяционной
    статистики по ключам; target-зависимые счётчики (fraud_rate, WOE, …) на
    pretrain не обновлять.
  - Фаза 2: проход train — обновление «чистых» marginals только по строкам,
    у которых event_id нет в train_labels; отдельные счётчики для fraud-показателей
    по правилам документа.

Выход:
  - Запись итоговых таблиц в `output/global_aggregates/` (или выбранный каталог)
    в формате Parquet.

Этот шаг выполняется до (или как обязательный префикс) полной сборки строк
обучающего датасета, чтобы `build_dataset` мог открыть готовые таблицы.


--------------------------------------------------------------------------------
4. Этап 2: изменения dataset_cpp
--------------------------------------------------------------------------------

При создании `output/full_dataset.parquet` (и при необходимости любых других
train-артефактов, которые несут те же признаки):

  - При старте загрузить с диска parquet-таблицы популяционных агрегатов
    (через Arrow C++), построить in-memory структуры для lookup по ключу:
    строка транзакции → ключ(и) → готовый набор числовых полей.

  - Для каждой записываемой строки датасета:
      * посчитать как сейчас признаки уровня пользователя/окна;
      * дописать колонки глобального блока из lookup (при отсутствии ключа —
        согласованный MISSING/backoff из спеки).

  - Порядок и имена колонок в выходном parquet должны совпадать с контрактом
    Python: `kFeatureNames` / `FEATURE_NAMES` (расширение списка в одном месте
    с обновлением C++ и Python).

Важно: на test-строках, если они когда-либо проходят через тот же бинарник для
локальной проверки, подставляются только замороженные таблицы (без включения
test в пересчёт агрегаторов). Для основного сабмита см. п.5.


--------------------------------------------------------------------------------
5. Этап 3: Python — инференс (submission и скрипты)
--------------------------------------------------------------------------------

Поведение:
  - При старте `submission/main.py` (или общий модуль `shared/…`) один раз:
      * прочитать те же parquet-файлы из `GLOBAL_CATEGORY_AGGREGATES_DIR`
        (или конкретные пути из `shared/config.py`);
      * для каждой оси ключа построить быстрый lookup: pandas Index + iloc,
        или dict по нормализованному ключу, или pyarrow Table + фильтр —
        исходя из объёма и профиля.

  - В точке, где для строки уже известны сырьевые поля (mcc_code, канал, …) и
    локальные фичи из `compute_features`:
      * добавить в словарь фич значения из глобальных таблиц по тем же правилам
        ключа и MISSING, что и в C++ при сборке датасета.

  - Итоговый `pd.DataFrame` для модели строится как сейчас:
    `MODEL_INPUT_FEATURES` в фиксированном порядке — глобальные колонки должны
    войти в этот список на тех же позициях, что и в parquet при обучении.

Контроль консистентности:
  - Один и тот же набор файлов агрегатов для: последней сборки train-датасета и
    для сабмита (версионирование по имени каталога или хешу в логе при сборке).


--------------------------------------------------------------------------------
6. Краткая последовательность команд (целевой workflow)
--------------------------------------------------------------------------------

  1) Собрать и запустить C++: построение глобальных агрегатов → запись в
     `output/global_aggregates/*.parquet`.

  2) Собрать и запустить `dataset_cpp/build_dataset`: чтение агрегатов + запись
     `output/full_dataset.parquet` с уже включёнными колонками популяции.

  3) `PYTHONPATH=. python training/main.py` — без join глобальных фич в Python.

  4) `PYTHONPATH=. python submission/main.py` — загрузка тех же parquet
     агрегатов + подмешивание к фичам строки перед предсказанием.


--------------------------------------------------------------------------------
7. Итог
--------------------------------------------------------------------------------

  - C++ пишет популяционные таблицы в Parquet (удобно для Arrow и Python).
  - dataset_cpp подмешивает их в обучающий датасет при генерации строк
    (быстрое обучение: всё в одном широком parquet).
  - Инференс на Python подтягивает тот же артефакт один раз и воспроизводит
    join по ключу построчно, сохраняя идентичность имён и порядка признаков
    с обучением.
