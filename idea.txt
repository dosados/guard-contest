Подробное описание агрегации фичей и подготовки входа модели
============================================================

Этот документ описывает фактическую логику из текущего кода проекта:
- сборка обучающего датасета: `dataset_cpp/build_dataset.cpp`;
- общая Python-логика агрегации/фич: `shared/parquet_batch_aggregates.py`, `shared/features.py`;
- обучение: `training/main.py`;
- инференс/сабмит: `submission/main.py`.


1) Общий принцип: "фичи считаются по истории ДО текущей строки"
---------------------------------------------------------------

Ключевая идея одинакова в C++ и Python:
- Для каждой транзакции клиента берется окно его предыдущих транзакций.
- Фичи считаются только по этому окну (текущая строка еще не включена).
- После вычисления фич текущая транзакция добавляется в окно.

Это устраняет утечку таргета/будущего в признаки.


2) Что такое окно и как выбирается его длина
--------------------------------------------

2.1 Окно в C++ (сборка train-датасета)
--------------------------------------

В `dataset_cpp/build_dataset.cpp`:
- хранится `deque` на клиента;
- физический максимум окна: `kWindowCap = 150` (старые записи выталкиваются из начала);
- для каждой train-строки выбирается "эффективная длина окна" `eff`:
  - сначала детерминированно сэмплируется целевое значение `W_target` в диапазоне [101..150];
  - распределение экспоненциальное: вес `(101 + k)` пропорционален `exp(-0.12 * k)`;
  - на практике чаще берутся более короткие окна, но в пределах 101..150;
  - затем `eff = min(stored, W_target)`, где `stored` = сколько уже транзакций накоплено в deque.

Итог:
- если история короткая (`stored < 101`) -> используется вся доступная история;
- если история длинная -> берется суффикс окна длиной `eff` (последние `eff` транзакций).

Важно: `W_target` детерминирован для пары `(event_id, customer_id)` (через хеш), то есть повторяем между запусками.

2.2 Окно в Python-инференсе (`submission`)
------------------------------------------

В `submission/main.py` + `shared/parquet_batch_aggregates.py`:
- агрегаты на pretest строятся с `unlimited_window=True` (без лимита по длине deque);
- дальше при проходе test окно каждого клиента продолжается от pretest;
- фичи также считаются "до update".

То есть на инференсе используется максимально полная история (pretest + уже пройденная часть test), а не cap=150.

2.3 Окно в Python-утилитах (если используются отдельно)
------------------------------------------------------

В `shared/dataset_settings.py`:
- `WINDOW_TRANSACTIONS = 150`;
- есть режим `DATASET_MODE = "window_50"` для ограничения окна 50, но по умолчанию `full`.

Это влияет на Python-пайплайны, которые пользуются `UserAggregates` без `unlimited=True`.


3) Фазы сборки обучающего датасета (C++)
---------------------------------------

Сборщик `build_dataset` для k = 1,2,3 подряд делает:
1. `pretrain_part_k` — только наполняет окна этой когорты; в датасет не пишет.
2. `train_part_k` — для каждой строки: фичи по окну до строки → запись в `output/full_dataset.parquet` → update окна.
3. Удаляет из map все `customer_id`, встреченные в этой паре файлов (когорты с разными k не пересекаются; см. `scripts/check_pretrain_train_customer_disjoint.py`).

Итоговый порядок строк в parquet — как при чтении только train: `train_part_1`, затем `train_part_2`, затем `train_part_3`. Pretrain по-прежнему разогревает историю перед своим `train_part_k`.


4) Какие признаки считаются и из чего
-------------------------------------

Имена признаков фиксированы (`FEATURE_NAMES`) и одинаково используются в train/inference.
Ниже логика по группам.

4.1 Денежная статистика по окну
-------------------------------

По `operaton_amt` из окна:
- `amount_to_median` = текущая сумма / медиана окна;
- `amount_zscore` = z-score по mean/std окна (std по sample stdev, минимум 2 значения);
- `is_amount_high` = 1, если текущая сумма > p95 окна;
- `amount_diff_prev`, `amount_ratio_prev` относительно последней суммы в окне;
- `trend_mean_last_3_to_10` = mean(last3) / mean(last10);
- `amount_percentile_rank` = доля сумм в окне <= текущей;
- `max_amount_last_24h`, `sum_amount_last_1h`, `sum_amount_last_24h` по подмножеству "за последние 24ч".

4.2 Частотные временные признаки
--------------------------------

По времени `event_dttm`:
- `transactions_last_10m`, `transactions_last_1h`, `transactions_last_24h`;
- `time_since_prev_transaction` (если не вычислилось, в C++ ставится -1);
- `hour`, `day_of_week`, `is_night_transaction`, `is_weekend`.

4.3 Новизна категорий / переключения
------------------------------------

На основе счетчиков внутри окна:
- `is_new_device` (пара os_type + device_system_version);
- `is_new_mcc`;
- `is_new_channel` (type + subtype/sub_type);
- `is_new_timezone`;
- `is_new_browser_language`;
- `is_new_device_tz_pair`.

Переключения относительно последней записи окна:
- `is_timezone_change`, `is_device_switch`, `is_mcc_switch`;
- `time_since_last_device_change`, `time_since_last_mcc_change`.

4.4 Сессии
----------

По `session_id` в окне:
- `transactions_in_session` (в коде: `n_sess + 1`, где `n_sess` уже в окне);
- `session_duration` (max(dttm)-min(dttm) для session_id);
- `session_mean_amount`.

4.5 Признаки на дельтах времени
-------------------------------

По последовательности времен в окне:
- `std_time_deltas`;
- `delta_1`, `delta_2`, `delta_3` (последние интервалы в секундах);
- `acceleration_delta_1_over_2` = delta_1 / delta_2;
- `std_delta_last_k`, `cv_delta_last_k` (k = последние до 10 дельт).

4.6 Категориальные числовые surrogate
-------------------------------------

Для `event_descr` и `mcc_code`:
- строка хешируется MD5;
- берется остаток `% 1_000_000`;
- получается float-число (`event_descr`, `mcc_code` фичи).

4.7 Interaction-фичи
--------------------

- `amount_zscore_x_is_new_device`;
- `amount_zscore_x_is_new_mcc`;
- `is_new_device_x_is_new_timezone`;
- `device_freq` = count(device_key in window) / eff.


5) Как заполняются "пустые места" и неопределенности
---------------------------------------------------

Логика смешанная: где-то NaN, где-то 0/1 или -1. Это важно, потому что модель видит именно это.

5.1 Когда нет истории/недостаточно данных
-----------------------------------------

Обычно ставится NaN:
- медиана/стд/отношения/ранги/часть дельт;
- `session_duration`, `session_mean_amount` без данных;
- `device_freq` при пустом окне.

Исключения:
- `time_since_prev_transaction` в C++ при отсутствии валидной пары времени -> `-1`;
- `max_amount_last_24h` если recent пуст -> 0;
- счетчики `transactions_last_*`, `sum_amount_last_*` -> 0;
- бинарные признаки по заполненности (`web_rdp_connection`, `phone_voip_call_state`) -> 0, если пусто.

5.2 Пустые/битые исходные поля
------------------------------

- Пустые строки trim-ятся и считаются пустыми.
- Некорректный `operaton_amt` -> как отсутствующий (в статистики не попадает, текущий amount может стать NaN).
- Некорректный `event_dttm` -> нет временных фич (часть станет NaN/0/-1 по правилам выше).
- Для channel subtype есть fallback:
  - сначала `channel_indicator_sub_type`;
  - если пусто/None -> `channel_indicator_subtype`.
- Для описания события:
  - сначала `event_descr`;
  - иначе `event_desc`.

5.3 Деление на ноль
-------------------

Во всех ratio используется epsilon (`1e-9`), чтобы не падать и получать конечные значения.


6) Target и sample_weight в train-датасете
------------------------------------------

В `build_dataset.cpp`:
- читается `data/train_labels.parquet` как map `event_id -> original_target(0/1)`;
- если event_id найден в labels:
  - в датасете `target = 1` (флаг "размеченный как подозрительный/важный");
  - `sample_weight = 5` если original_target=1, иначе `2`;
- если event_id не найден:
  - `target = 0`;
  - `sample_weight = 1`.

То есть таргет в train-матрице бинаризован по факту наличия event_id в labels, а исходный 0/1 в labels влияет только на вес.


7) Что происходит в training (подготовка входа модели)
------------------------------------------------------

`training/main.py` обучает модели из `output/full_dataset.parquet` (streaming-батчами):
- определяет список фич через `resolve_model_input_columns`:
  - обязаны присутствовать все `FEATURE_NAMES`;
  - порядок строго фиксирован `MODEL_INPUT_FEATURES`.
- в каждом батче:
  - `X` = выбранные фичи, приводятся к numeric, dtype float32;
  - `y` = `target`, int32;
  - `w` = `sample_weight`, float32.

Дополнительно веса ремапятся (`shared/config.py`):
- 1 -> 1
- 2 -> 5
- 5 -> 10

Валидация по времени:
- по `event_dttm` считается day-cutoff;
- train: строки с датой < cutoff;
- val: остальные (последние ~15% по времени).

Модели (XGBoost/LightGBM) обучаются побатчево, по одному бустинг-раунду на батч.


8) Что происходит в submission (инференс)
-----------------------------------------

`submission/main.py` строит признаки тем же `compute_features` из `shared/features.py`:

1. Pretest-фаза:
   - читается `pretest.parquet`;
   - для каждого клиента наполняется история (`UserAggregates`) без лимита окна.

2. Test-фаза:
   - строки `test.parquet` идут по порядку;
   - для каждой строки берется агрегат клиента;
   - считаются фичи по истории ДО строки;
   - затем строка добавляется в агрегат.

3. Предсказание:
   - формируется DataFrame из `feature_matrix`;
   - берутся ровно `MODEL_INPUT_FEATURES` в том же порядке, что на обучении;
   - модель выдает probability, которая переводится в logit (`log(p/(1-p))`);
   - пишется `output/submission.csv` с колонками `event_id,predict`.

Это гарантирует консистентность: на инференсе входные признаки агрегируются по той же логике "до update" и в том же порядке, что использовался для обучения.


9) Ключевой итог по логике агрегации
------------------------------------

Данные для модели формируются как "снимок состояния клиента на момент текущей транзакции":
- используются только прошлые транзакции клиента;
- окно истории ограничено правилами конкретного этапа (train C++: cap150 + sampled eff, submission: unlimited);
- пропуски и нестабильные вычисления обрабатываются детерминированно (NaN/0/-1/epsilon);
- список и порядок признаков единый от train до inference.

Именно это и определяет, как агрегируются входные данные модели во всем пайплайне.
